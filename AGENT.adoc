== Projet

**Nom** : magic_stick (environnement portable standardise sur cle USB bootable)
**But** : Fournir un environnement de dev/diagnostic complet, bootable sur tout PC, independant de l'OS hote
**Stack** : Bash, live-build (Debian/Ubuntu ISO customization), xorriso, syslinux/GRUB

=== Cas d'usage

* **Formation informatique** : les apprenants ont tous la meme cle, prets a bosser des le 1er jour
* **Technicien terrain** : diagnostique reseau/FTTH quel que soit l'ordi
* **Dev nomade** : retrouver sa stack complete sur n'importe quel PC
* **Equipe technique** : environnement standardise partage, pas de temps perdu en config

=== Architecture des partitions

[cols="1,2,3"]
|===
| Partition | Taille | Role

| System A (lecture seule)
| ~8 Go
| Systeme live Xubuntu actif (boote)

| System B (lecture seule)
| ~8 Go
| Systeme live Xubuntu de secours (update atomique)

| Persistence (lecture/ecriture)
| Reste de la cle
| Donnees utilisateur + config logiciels
|===

**Principe A/B** : Quand on met a jour la partition systeme, on ecrit sur la partition inactive (A ou B), on inverse le flag de boot, et au prochain demarrage on est sur la nouvelle version. Si ca plante, on reboot sur l'ancienne. La partition Persistence n'est JAMAIS touchee par les mises a jour systeme.

=== Secrets et tokens — INTERDICTION DE LOG

**PRINCIPE** : Les tokens API, mots de passe, et identifiants credentials NE DOIVENT JAMAIS apparaitre dans la sortie terminal de l'agent. L'utilisateur peut streamer son terminal (Discord, Twitch, etc.). Tout token visible dans le terminal est compromis.

**INTERDICTIONS** :

* **NE JAMAIS** passer un token en argument visible d'une commande (ex: `GH_TOKEN=xxx gh run list` affiche le token dans l'historique shell)
* **NE JAMIS** afficher le contenu d'un fichier contenant des tokens (site.yml, .env, credentials, etc.) si le token peut etre visible dans la sortie
* **NE JAMAIS** utiliser `echo`, `cat`, ou tout commande qui affiche des tokens dans le terminal

**Procedure OBLIGATOIRE pour utiliser des tokens** :

. Utiliser `gh auth login --with-token` avec le token via **stdin** (pipe ou redirect), jamais en argument visible
. Utiliser des variables d'environnement dans un sous-shell : `(< token gh auth login --with-token)` plutot que `GH_TOKEN=token gh ...`
. Verifier que `gh auth status` fonctionne apres configuration
. Ne **jamais** lire puis afficher le contenu d'un fichier de credentials (site.yml, .env, etc.)

**Lecon de la Session 008 (magic_stick)** : L'agent a affiche le contenu de `site.yml` contenant un token GitHub `gho_...` dans le terminal, puis a utilise `GH_TOKEN=xxx gh run list` qui affiche le token dans l'historique shell. L'utilisateur streamait son terminal sur Discord. Token compromis en 1 seconde.

=== Fichiers de configuration — REGLE ABSOLUE DE SURETE

**PRINCIPE** : Tout fichier de configuration (YAML, JSON, XML, TOML, properties, .env, etc.) peut contenir des donnees sensibles non versionnees. Modifier un tel fichier sans precaution met en danger l'integrite du travail du developpeur.

**INTERDICTIONS** :

* **NE JAMAIS ecraser** un fichier de configuration existant avec un `Write` complet si on peut faire un `Edit` partiel
* **NE JAMAIS remplacer** des valeurs sensibles existantes (tokens, mots de passe, cles API, URLs de production) par des valeurs factices ou placeholders
* **NE JAMAIS modifier** un fichier sans avoir **d'abord** verifie s'il est dans `.gitignore` (`git check-ignore <fichier>`) — un fichier ignore n'a pas de `git restore` possible

**Procedure OBLIGATOIRE** avant modification d'un fichier de configuration :

. `git check-ignore <fichier>` : Verifier si le fichier est versionne
. `git ls-files -- <fichier>` : Confirmer s'il est tracke
. Si le fichier **n'est pas tracke** (pas dans git) : **CREER UNE SAUVEGARDE** avant toute modification (`cp <fichier> <fichier>.bak`)
. Si le fichier contient des valeurs sensibles : **les PRESERVER intactes**, ne jamais les remplacer par des placeholders
. Privilegier **l'edit partiel** (modification ciblee) plutot que **l'ecriture complete** (ecrasement total)
. Apres modification : **verifier** que les valeurs sensibles sont toujours presentes

**Lecon de la Session 2 (bakery-plugin)** : Ecraser un site.yml non versionne avec des valeurs factices a detruit des tokens GitHub reels. Sans sauvegarde git (fichier ignore), la restauration aurait ete impossible sans une copie externe. Cela aurait bloque definitivement le developpeur si aucun fallback n'etait disponible. **Ce n'est pas une question de viser un fichier site.yml — c'est un comportement de surete qui ne met pas en danger le travail du developpeur.**

=== Fichiers personnels — Interdiction de modifier

_L'agent NE DOIT JAMAIS_ modifier, ecraser ou supprimer un fichier personnel de l'utilisateur (`~/.gitconfig`, `~/.bashrc`, `~/.ssh/config`, `~/.zshrc`, etc.) _SANS permission explicite de l'utilisateur_.

**PRINCIPE** : Les fichiers `~/.*` appartiennent a l'utilisateur. Leur modifier peut casser le workflow, les credentials, ou les preferences de l'environnement. L'environnement de travail de l'utilisateur est en dehors du scope du projet.

**INTERDICTIONS** :

* **NE JAMAIS** utiliser `Write` ou `Edit` sur un fichier `~/.*` ou `~/*` qui n'est pas dans le repertoire de travail du projet
* **NE JAMAIS** executer de commande touchant `~/.gitconfig`, `~/.bashrc`, `~/.ssh/*`, `~/.zshrc`, etc. meme avec l'intention de "reparer" ou "configurer"
* **Toujours rester dans le working directory** du projet actuel

_Lecon Session 033 (magic_stick)_ : L'agent a tente de modifier `/home/cheroliv/.gitconfig` pour changer le `user.name` de `opencode` vers `cheroliv`. Sans permission explicite. Modifier les configs utilisateur hors projet est un comportement catastrophique qui peut corrompre l'environnement de travail.

=== Methodologie de Travail

* **1 fichier a la fois** : Modifier un seul fichier avant de passer au suivant
* **Validation immediate** : Verifier la syntaxe bash apres chaque modification (`bash -n script.sh`)
* **Contexte leger** : Maximum 50k tokens EAGER (~3000 lignes)

== Composants Cles

* **live-build** : Outil Debian/Ubuntu pour construire des ISOs live customisees
* **Partition A/B** : Systeme de double partition pour mises a jour atomiques
* **Persistence** : Partition ext4 avec `persistence.conf` pour donnees utilisateur
* **Scripts chroot** : Installation de Ollama, Docker, SDKMAN, etc. dans le chroot live-build
* **GRUB/syslinux** : Boot loader configure pour booter sur A ou B
* **Convergence chroot/Dockerfile** : Les versions des outils CLI sont centralisees dans `scripts/lib/versions.sh`. Le hook chroot `060-install-chroot-software.chroot` source ce fichier. Le `docker/magic-stick-cli/Dockerfile` declare des ARG portant les memes valeurs que `versions.sh`. La mise a jour d'une version doit etre faite en parallele dans les deux fichiers :
** `scripts/lib/versions.sh` (maitre)
** `docker/magic-stick-cli/Dockerfile` (ARG, a maintenir aligne)
** `config/live-build/hooks/060-install-chroot-software.chroot` (deja source `versions.sh`, pas de valeurs hardcodees)

== Session Procedure

=== Debut de Session

1. Lire `PROMPT_REPRISE.adoc` pour la mission en cours
2. Lire `.agents/INDEX.adoc` pour les regles absolues
3. Charger les fichiers ESSENTIALS si l'EPIC est concernee

=== Fin de Session

**Mots-cles declencheurs** : "nouvelle session", "je quitte", "session terminee", "a plus tard", "on arrete la"

**Rotation backup** : Mots-cles explicites — "rotation backup", "backup rotation", "lance la rotation backup" — declenchent une rotation independante (hors fin de session).

**Procedure obligatoire en 6 etapes** :

1. **Creer l'archive LAZY** : `.agents/sessions/{N}-{type}-{sujet}.adoc`
2. **Mettre a jour PROMPT_REPRISE.adoc** — Mission N+1 (max 70 lignes)
3. **Mettre a jour SESSIONS_HISTORY.adoc** : `.agents/SESSIONS_HISTORY.adoc` — Ajouter la session terminee
4. **Mettre a jour INDEX.adoc** : `.agents/INDEX.adoc` — Date, session en cours N+1
5. **Mettre a jour TEST_COVERAGE_ANALYSIS.adoc** (si tests modifies)
6. **Mettre a jour COMPLETED_TASKS_ARCHIVE.adoc** : `.agents/archives/COMPLETED_TASKS_ARCHIVE_{mois}.adoc`

**Aucune etape ne peut etre marquee OK si le fichier n'a pas ete effectivement modifie et verifie.**

==== Capteur de Rotation Backup (etape 7 optionnelle)

**Declenchement automatique** — Apres l'etape 6, avant la checklist, l'agent verifie :

. `N % 10 == 0` (session multiple de 10)
. **OU** les fichiers EAGER cumules depassent **500 lignes** (`wc -l .agents/INDEX.adoc .agents/archives/COMPLETED_TASKS_ARCHIVE_*.adoc PROMPT_REPRISE.adoc`)

Si l'une des conditions est vraie, l'agent **execute l'etape 7** au sein de la procedure de fin de session :

. Creer le dossier `.agents/backup/Y{annee}-sessions-{plage}/` (plage = sessions N-5* et anterieures)
. Y copier **integralement** INDEX, SESSIONS_HISTORY, COMPLETED_TASKS_ARCHIVE et les fichiers sessions/ concernes
. Tronquer les fichiers actifs aux **10 dernieres sessions** (sliding window)
. Ajouter la localisation de la nouvelle vague dans INDEX.adoc (`_Localisation active_`)

La checklist s'affiche ensuite **apres** cette etape, avec `[✅] 7. Backup rote (si applicable)`.

=== Mécanisme de Backup — Hot/Warm/Cold Storage

* **Hot (EAGER)** : 10 dernieres sessions dans INDEX, PROMPT_REPRISE N+1, sessions N-1 et N, SESSIONS_HISTORY recent
* **Warm (LAZY)** : références techniques, procédures, agents spécialisés
* **Cold (`backup/`)** : Sessions N-10 et antérieures — copie intégrale, sans transformation ni réindexation
* **Cold (`encyclopedies/`)** : Connaissance thématique exhaustive (méthodologie, patterns) — jamais chargé automatiquement, humain seulement
* **Règle de rotation** : Tous les ~10 sessions, déplacer le paquet de fichiers agents (INDEX, SESSIONS_HISTORY, COMPLETED_TASKS_ARCHIVE, sessions/) tel quel dans `backup/`
* **Stockage** : `.agents/backup/Y{année}-sessions-{plage}/` — structure identique à `.agents/` actif
* **Principe** : Aucune donnée supprimée, tout relocalisé. L'agent ne charge jamais `backup/` ni `encyclopedies/`. L'humain seul décide de la distillation.
